Baker

  • Idea
  • Why
  • Interactions
  • Ingredients
  • Events
Now it looks like I have not really talked about Baker yet. But I actually have.
If you understand the idea, the need for it. And terms like Interaction, Ingredients and Events
Then you are actually almost there.
Let's again take a look at the ChopCarrot Interaction
The only difference with Baker compared to the story so far, is that Baker handles based on Events.
An Interaction will not return an Ingredient, but it will return an Event.
And that Event can hold Ingredients. Like normal events in Java van also hold data.
And again as in normal Java, an event is not limited to just one Ingredient
Its also possible for Interactions to return different Events.
Like we could have cut our finger and abort the whole cooking instead of providing chopped carrots
Programming based on Events is handy because it promotes loose coupling of the code.
You do not really care who produced and Event. You just want to do a certain Interaction based on certain Events
Let the real Baker coding begin.
First you will need to add baker to project. Which you can do via SBT
Or mvn.
https://ing-bank.github.io/baker/getting-started
Which is something no one is ever going to remember.
So I will just link to our documentation website at the end of the slides
For every Interaction you will need to make its Interface.
There you will tell which Ingredients are needed for the Interaction
And which Events they can return.
CarrotChopped and FingerCut are both instances of ChopCarrotOutcome
The Events are just standard java interfaces and classes.
We define an interface ChopCarrotOutcome, and two classes that implement ChopCarrotOutcome
CarrotChopped and FingerCut
You will be creating this simple Interaction interfaces and Events for all your Interactions
And then you can create what we call a Recipe, where you specify that the recipe "spaghetti"
consists of all these Interactions.

Sensory Events

  • Allow us to interact with our Recipe
  • Event to trigger an Interaction
  • Event to provide Ingredients
There is one thing we have not really discussed yet. And that is how to interact with our recipe.
Ingredients like Carrot and Knife are not available out of the blue. They need to be provided.
We need a way to interaction with our Recipe. To trigger and Interaction or to provide Ingredients
We do that via what we call "Sensory Events"
For our spaghetti demo I made two Sensory Events. ShoppedGroceries to provide the eatable wares.
And GatherKitchenware to provide us with the tools.
You will need to specify those Sensory Events as "entry point" Events to the Recipe.
Now at this moment our Recipe is like the blueprint of our process.
It is not actually able to run anything because we only gave it Interfaces and not real implementation.
But these Interfaces of Interactions and Events is all Baker need to determine the order of execution.
To orchestrate the process.
We could now Compile our recipe. Which will figure out how all Interactions etc are related.
And then ask for a .dot representation of the recipe.
Which is something that tools like Graphivz understands and can transform in this graph.
Now the text in there is not really relevant for the demo.
What is relevant is that we did not specify the order of execution.
Baker figures that our based on the given Interactions, Events and Ingredients.
Non developers can look at this graph, which is created from the official code and determine if things are not happening in order as they excepted.
Which means there is a fault in the thinking. Which now probably just require some changes in the recipe. And not in the big mess of orchestration code
To finish the pieces. You will be needing to create implementations of your Interactions.
Which is just a class that implements the Interaction interface and overwrite the apply method with the real implementation
Now to actually execute your recipe. To run your process.
You will be needing a Baker instance
Which is just new JBaker for Java
Then you can add your Implementations of your Interactions to Baker.
And also compile your recipe and give that to Baker.
Baker should now be able to match the implementations to the recipe blueprint.
And will know what to execute if certain Ingredients or Events are available.
Baker will also give you back a recipe id which you need to use to to interact with that recipe.
